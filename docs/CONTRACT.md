# Контракт системы (Этап 0) — источник правды

Документ фиксирует договорённости до кода. Изменения — только осознанно, с версией внизу.

---

## 1. Платформы

| Роль | ОС | Примечание |
|------|-----|------------|
| Продакшен (камера HikRobot, финальная сборка) | **Windows** | Официальный SDK HikRobot, named pipes, при необходимости JNA/JNI под Java. |
| Разработка | **Linux** | Сборка/тесты с **заглушкой камеры** (файл, синтетика, или другой источник); поведение пайплайна и форматов данных совпадает с Windows. |

**Правило:** один и тот же **логический** контракт кадра и сообщений на обеих ОС; отличается только слой захвата (SDK vs stub).

---

## 2. Камеры и параллелизм

- **Камер: 5.** Каждая камера — **отдельный процесс worker** (как в исходном промпте): изоляция падений, свой кольцевой буфер.
- **«По кругу ведра»** трактуем так: у каждого worker **своё кольцо слотов** (ring) фиксированной глубины; при перегрузе — **перезапись самого старого слота** или явный **drop** с инкрементом счётчика (конкретная политика задаётся на этапе реализации кольца, но **без** неограниченных очередей кадров).

Оркестратор (Java) знает **camera_id** 0…4 и маршрутизацию команд/конфига по процессам.

---

## 3. Кадр (пиксели)

| Параметр | Значение |
|----------|----------|
| Разрешение | **2448 × 2048** |
| Порядок пикселей | Строки слева направо, сверху вниз |
| Формат по умолчанию | **BGR, 8 бит на канал, 3 канала** (`uint8`, шаг строки = 2448×3 при отсутствии выравнивания; если SDK даёт выровненный stride — в контракт выносится отдельное поле `stride_bytes` в метаданных слота) |

Сырой кадр **не** передаётся внутри JSON. Между компонентами — **указатель на слот / shared memory / дескриптор региона** + метаданные (ниже).

---

## 4. Доступ Java к изображению (геометрия)

Требование: **Java должна иметь доступ к текущему кадру** (алгоритмы геометрических дефектов на Java).

**Запрещено в горячем пути:** класть полный кадр в JSON/Base64 для внутреннего вызова Java.

**Допустимые варианты** (выбрать один основной на этапе интеграции; остальные — запасной):

1. **Общая память (рекомендуется для минимальных копий):** нативный слой (C или маленькая DLL) выделяет **именованный memory-mapped регион** на кадр или на слот кольца; Java читает через **JNA / JNI / MemorySegment (JDK 22+)** по согласованному layout + `frame_id` / `slot_index`.
2. **Передача дескриптора:** worker пишет кадр в регион, по **named pipe** в Java уходит только **JSON-метаданные**: `{ "camera_id", "frame_id", "width", "height", "format", "shm_name" | "offset", "stride_bytes" }` без пикселей.
3. **Запасной путь для отладки на Linux:** тот же контракт метаданных + mmap файл в `/dev/shm` — семантика как у shared memory на Windows.

**Синхронизация:** кадр считается «готов к чтению Java» только после **одного** примитива «commit» (версия слота / sequence lock / event), чтобы не читать полузаписанный буфер.

## 5. Python: вход / выход (контракт полей)

**Назначение:** детекция на Python получает **view** на буфер кадра (без сериализации) + **ROI**.

### 5.1 Вход детектора (логический; не обязательно один JSON на кадр)

- Кадр: по ссылке на буфер (см. §3–4).
- **ROI для Python** — только то, что нужно детектору, в **пиксельных координатах** относительно полного кадра:

```json
{
  "camera_id": 0,
  "frame_id": 18446744073709551615,
  "rois": [
    { "name": "main", "x": 0, "y": 0, "width": 2448, "height": 2048 }
  ]
}
```

Имена и набор ROI расширяются конфигом; координаты целые, `0 <= x`, `0 <= y`, `x+width <= 2448`, `y+height <= 2048`.

### 5.2 Выход детектора (результат для оркестратора / следующих шагов)

Поля **только про ROI** (числовые/структурные результаты по регионам без обязательной передачи целого изображения):

```json
{
  "camera_id": 0,
  "frame_id": 18446744073709551615,
  "detector_id": "v1",
  "roi_results": [
    {
      "name": "main",
      "anomaly_score": 0.0,
      "status": "OK",
      "details": {}
    }
  ]
}
```

- `details` — произвольный вложенный объект под конкретный детектор (пороги сравнения на стороне Python **могут** быть в конфиге; в **этот** JSON по договорённости этапа 0 — только результаты по ROI, не обязательно дублировать все пороги).

---

## 6. Java: вход для геометрии (ROI + допуски)

Java получает **доступ к пикселям** по §4 и отдельно — **параметры допусков** и ROI для геометрического блока (в духе твоего `InspectionRequest`):

```json
{
  "camera_id": 0,
  "frame_id": 18446744073709551615,
  "pixels_to_mm": 0.01,
  "main_roi": { "x": 0, "y": 0, "width": 2448, "height": 2048 },
  "joint_roi": null,
  "wrinkles_roi": null,
  "max_shift_mm": 0.5,
  "max_rotation_deg": 1.0,
  "max_concentricity_mm": 0.2,
  "max_joint_defect_mm": 0.3,
  "max_wrinkles_score": 0.05
}
```

- Поля `*_roi` и пороги — как в прикладной геометрии; `null` для опциональных ROI допустим.
- Привязка к кадру: обязательны `camera_id` + `frame_id` + ссылка на тот же буфер, что и у метаданных shared memory / pipe (§4).

Выход Java (результат геометрии) на этапе 0 не жёстко фиксируем полным списком полей — но **обязательны**: `frame_id`, итоговый `PASS`/`FAIL`, числовые метрики, при необходимости `debug_image` только если явно включено в конфиге (отдельный канал или сжатый артефакт, не основной hot path).

## 7. Робот (манипулятор)

**Статус:** протокол **не выбран** (Modbus, PROFINET, дискретные выходы, TCP-шлюз и т.д.).

**Зафиксировано заранее:**

- После готовности **вердикта** по `frame_id` сигнал на робота и обновление клиента идут **параллельно** (два независимых исходящих пути; очереди **ограниченной** глубины; задержка UI не блокирует команду робота).
- В контракт робота позже добавится: формат пакета/регистров, таймаут, маппинг `PASS`/`FAIL` → действие.

## 8. Клиент (Electron на том же ПЛК)

**Контекст:** UI и бэкенд на **одной машине**; нужен **максимально быстрый** обмен при типичной схеме: превью + JSON результата, общий `frame_id`.

### 8.1 Рекомендация по умолчанию

**WebSocket (`ws`) на `127.0.0.1`:**

- Сообщения **бинарные** — JPEG (или WebP) превью; короткий **бинарный или текстовый** заголовок с `frame_id` и флагами (или отдельное маленькое JSON-сообщение сразу после бинарного кадра в том же канале по согласованному framing).
- Плюсы: нативно в Electron/Node, один порт, мультиплексирование, без тяжёлого HTTP-стека на каждый кадр.
- На localhost задержка обычно **достаточна** для HMI при разумном FPS превью (не каждый полный 2448×2048 без сжатия).

### 8.2 Если понадобится ещё меньше накладных расходов (Windows)

**Named Pipe** (`\\.\pipe\...`) с **length-prefixed** кадрами: Node подключается через `net.connect`; меньше оверхеда, чем у HTTP. Минус: чуть больше возни с фреймингом и переподключением.

### 8.3 Не для первого этапа без необходимости

- Полный RAW 2448×2048×3 по сокету каждый кадр — высокая нагрузка на CPU/шину; для UI обычно **масштаб + JPEG**.
- **Shared memory напрямую в Electron** — возможно через native-addon; оставить как оптимизацию, если WebSocket на localhost станет узким местом.

**Итог этапа 0:** протокол клиента — **WebSocket на localhost с бинарными кадрами + метаданные по `frame_id`**; при необходимости смена на Named Pipe без смены логики вердикта и fan-out.

---

## 9. Сквозная идентификация

- **`camera_id`:** 0…4.
- **`frame_id`:** монотонно растущий **uint64** на камеру (или глобально уникальный — решается при реализации; в сообщениях всегда явное поле).

Все потребители (Python, Java, клиент, робот) коррелируют данные только по **`camera_id` + `frame_id`** + при необходимости `timestamp_ns` (опционально).
